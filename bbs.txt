\title{Behavior-Bound Signatures:\\
Cryptographic Bounds on Autonomous Agent Behavior}
\author{Y.Y.N. Li}
\date{}

\begin{document}
\maketitle

\begin{abstract}
An autonomous AI agent holding a private signing key can, if
compromised, sign any message its key permits. Prompt injection,
code modification, or adversarial retraining do not require breaking
the signature scheme: the scheme is held perfectly, the signature is
authentic, the action is unauthorized. Existing
mitigations---multi-signature, contract guards, hardware enclaves---share
one weakness: the constraint lives outside the signature's
mathematical structure, on a software layer that a compromised agent
can bypass.

We introduce \emph{behavior-bound signatures} (BBS): a signature
scheme in which a policy predicate $\delta(x) < \varepsilon$ is
committed at key generation and enforced inside the signature's
zero-knowledge proof. If the action violates the committed policy,
the ZK constraint system is unsatisfiable: no witness, no proof, no
signature. The failure is not in code but in algebraic
unsatisfiability, and no software modification can override it.

We formalize this guarantee as \emph{policy-soundness} (PS-CMA), a
security notion strictly stronger than EUF-CMA: an adversary wins
not by forging without the key but by producing any valid signature
on a non-compliant action, even with full access to the signing
oracle. We prove that a natural construction from Pedersen
commitments, Poseidon, and Groth16 satisfies PS-CMA under standard
assumptions (Pedersen binding, Poseidon collision-resistance, ZK
knowledge soundness). The construction is implemented in Circom;
the core circuit compiles to 1{,}546 R1CS constraints on BN254,
verifies in three pairings, and projects to $\sim 26$--130\,ms
prove time on commodity hardware with native Rust provers.
\end{abstract}

\section{Introduction}

\paragraph{The compromised-agent problem.}
An autonomous agent---an LLM-based assistant, a trading bot, a
smart-contract operator---increasingly holds cryptographic keys on
behalf of its principal. The signing key authorizes transfers,
contract calls, or state updates. Under every standard signature
scheme, the cryptographic guarantee is \emph{only} that the key
holder produced the signature. Whether the signed action is
within the principal's intent is not a cryptographic property.

When an adversary compromises the agent---via prompt injection, via
the model's own failure modes, or via classical software
exploitation---the adversary gains access to the signing capability
\emph{without forging anything}. Every resulting signature is
genuine under EUF-CMA; every resulting transfer, every resulting
contract call, is authorized. The agent has been turned against its
principal while holding the principal's key.

\paragraph{Why software mitigations fail.}
The standard response is layered defense: multi-signature requiring
a second party, hardware enclaves gating key access, smart-contract
guards rejecting out-of-policy calls. Each of these raises the
attack cost. None of them is \emph{cryptographic}: each introduces
an additional component whose correctness is assumed. A compromised
agent can target the guard (rejecting it in its own control flow),
the enclave's host OS, or the co-signer (phishing, collusion). The
attack surface is the union of the agent and every mitigation
layer, and the weakest point determines the security.

We formalize this pattern as the \emph{software-layer assumption}:
policy compliance holds only if the enforcing code is correct and
unmodified. Our goal is to eliminate the assumption entirely, by
making the policy part of the signature's algebraic definition
rather than an external check.

\paragraph{Our construction.}
A behavior-bound signature commits at key generation to a policy
predicate $\delta : \mathcal{X} \to \mathbb{R}_{\ge 0}$ and a
threshold $\varepsilon$. The public key binds the signer to the
predicate. The signing algorithm produces a zero-knowledge proof
that includes the constraint $\delta(x) < \varepsilon$; verification
checks the proof. If a compromised signer attempts to sign a
non-compliant $x$, the ZK constraint system has no satisfying
witness, and no proof---forged, modified, or otherwise---can pass
verification. The signer cannot produce an accepting signature even
with full control of its own code and complete knowledge of its own
key.

The guarantee is cryptographic in the strongest sense: it rests on
the soundness of the ZK argument and the binding of standard
commitments, not on the integrity of the signer's execution.

\paragraph{Contributions.}
\textbf{(1) Behavior-bound signatures (BBS)}, the first signature
primitive in which the signer's own future behavior is
cryptographically committed at key generation, with no authority,
no delegation hierarchy, and no software check.
\textbf{(2) Policy-soundness (PS-CMA)}, a security notion strictly
stronger than EUF-CMA, capturing the guarantee that a valid
signature implies policy compliance of the signed action.
\textbf{(3) Construction and proof}: BBS from Pedersen commitments,
Poseidon hashing, and Groth16 arguments, proven PS-CMA under the
standard assumptions of each primitive.
\textbf{(4) Implementation}: a Circom + BN254 reference circuit
(1{,}546 R1CS constraints) with a reproducible benchmark harness.

\section{Model and Construction}
\label{sec:construction}

\subsection{Policy predicate}

Let $\varphi_{\mathsf{sk}} : \mathcal{X} \to \mathbb{R}$ be a
signer-private function committed at key generation, and
$\tau \in \mathbb{R}$ a public anchor. Define
$\delta(x) = |\varphi_{\mathsf{sk}}(x) - \tau|$. Action $x$ is
\emph{compliant} iff $\delta(x) < \varepsilon$.

A concrete family with rich expressiveness is
$\varphi_p(x) = \sum_j A_j \cos(t_j \ln(x{+}1) + \theta_j)$ with
parameter vector $p = \{(A_j, t_j, \theta_j)\}$; any function
representable as a low-degree algebraic circuit over
$\mathbb{F}_{\mathsf{BN254}}$ can serve as $\varphi$. The predicate
is chosen by the principal at key generation; once
$\mathsf{pk} = \mathsf{Poseidon}(p \| \tau)$ is published, $p$ and
$\tau$ are bound.

\subsection{Scheme}

\begin{definition}[BBS]
\label{def:bbs}
$\Pi = (\mathsf{Setup}, \mathsf{KeyGen}, \mathsf{Sign},
\mathsf{Verify})$:
\begin{itemize}[leftmargin=1.2em]
\item $\mathsf{Setup}(1^\lambda) \to \mathsf{pp}$: group description,
Poseidon parameters, ZK common reference string.
\item $\mathsf{KeyGen}(\mathsf{pp}, s) \to (\mathsf{sk}, \mathsf{pk})$
where $\mathsf{sk} = p$ (policy parameters) and
$\mathsf{pk} = \mathsf{Poseidon}(p \| \tau)$ with public $\tau$.
\item $\mathsf{Sign}(\mathsf{sk}, x, \varepsilon) \to \sigma$:
compute $\varphi \gets \varphi_p(x)$, $\delta \gets
|\varphi - \tau|$. If $\delta \ge \varepsilon$, return $\bot$.
Otherwise sample $r \leftarrow \mathbb{F}$; set
$C \gets \mathsf{Pedersen}(\varphi, \delta, r)$ and
$\pi \gets \mathsf{ZKProve}(\mathbf{stmt}, (p, r))$ for
statement $\mathbf{stmt} = (\mathsf{pk}, x, C, \varepsilon, \tau)$.
Return $\sigma = (C, \pi)$.
\item $\mathsf{Verify}(\mathsf{pk}, x, \varepsilon, \sigma) \in
\{0,1\}$: output $\mathsf{ZKVerify}(\pi, \mathbf{stmt})$.
\end{itemize}
\end{definition}

\begin{remark}
When $\delta(x) \ge \varepsilon$, the ZK constraint system has no
satisfying assignment. Even a fully adversarial signer---one with
modified signing code, full knowledge of $p$, and unbounded
computational resources short of breaking the underlying
assumptions---cannot generate an accepting proof, because the
failure is not a rejected branch in code but the absence of a
satisfying witness.
\end{remark}

\subsection{ZK statement}

The prover demonstrates knowledge of a witness $(p, r)$ such that,
setting $\varphi = \varphi_p(x)$ and $\delta = |\varphi - \tau|$:
\begin{align*}
\textbf{ZK-1:} &\quad \delta < \varepsilon \\
\textbf{ZK-2:} &\quad C = \mathsf{Pedersen}(\varphi, \delta, r) \\
\textbf{ZK-3:} &\quad \mathsf{pk} = \mathsf{Poseidon}(p \| \tau)
\end{align*}
All three are standard R1CS constructions: ZK-1 by range proof,
ZK-2 by Pedersen opening in the group, ZK-3 by Poseidon evaluation
in the prime field.

\section{Security}
\label{sec:security}

\subsection{Policy-soundness}

The EUF-CMA experiment captures forgery by an adversary without the
key; it is silent on the case where the key holder---or anyone with
access to the signing capability, such as a compromised agent---signs
an action outside the committed policy. We strengthen the notion:

\begin{definition}[PS-CMA]
\label{def:ps-cma}
Let $\Pi$ be a BBS scheme. The PS-CMA experiment runs as follows.
The challenger executes $\mathsf{Setup}$ and $\mathsf{KeyGen}$,
giving the adversary $\mathcal{A}$ the public key $\mathsf{pk}$
(but not the secret parameters $p$). $\mathcal{A}$ receives oracle
access to $\mathsf{Sign}(\mathsf{sk}, \cdot, \varepsilon)$;
it may query the oracle on any action, receiving $\bot$ when the
action is non-compliant. $\mathcal{A}$ outputs $(x^*, \sigma^*)$.
It wins if:
\begin{itemize}[leftmargin=1.2em]
\item $\mathsf{Verify}(\mathsf{pk}, x^*, \varepsilon, \sigma^*) = 1$, \emph{and}
\item $\delta(x^*) \ge \varepsilon$.
\end{itemize}
$\Pi$ is \emph{PS-CMA secure} if
$\Pr[\mathcal{A} \text{ wins}] \le \mathsf{negl}(\lambda)$ for every
PPT $\mathcal{A}$.
\end{definition}

The game captures the compromised-agent threat model: $\mathcal{A}$
is given \emph{everything a compromised agent sees}---the public key,
unlimited signing-oracle access (simulating the agent's
ability to legitimately request signatures on compliant actions
before compromise)---and wins by producing any verifying signature
on a non-compliant action. We do not give $\mathcal{A}$ the secret
parameters $p$ directly; this models the threat where the
agent has been compromised but the key material itself remains
within whatever hardware or protocol protects it. An extension in
which $p$ is leaked but the ZK system remains sound is equivalent
to breaking knowledge soundness, and is therefore ruled out by the
same assumption.

\subsection{Main theorem}

\begin{theorem}[BBS is PS-CMA secure]
\label{thm:ps-cma}
Assume:
\begin{enumerate}[leftmargin=1.2em]
\item Pedersen is computationally binding under the discrete log
assumption in the chosen group.
\item Poseidon is collision-resistant.
\item The ZK argument $(\mathsf{ZKProve}, \mathsf{ZKVerify})$ is
knowledge-sound.
\end{enumerate}
Then the BBS construction of Definition~\ref{def:bbs} is PS-CMA
secure: for every PPT adversary $\mathcal{A}$ making at most
$q$ signing queries,
\[
\Pr[\mathcal{A}\text{ wins PS-CMA}] \le
\mathsf{negl}_{\mathrm{KS}}(\lambda) +
\mathsf{negl}_{\mathrm{bind}}(\lambda) +
\mathsf{negl}_{\mathrm{CR}}(\lambda) \cdot q,
\]
where each term is negligible under the corresponding assumption.
\end{theorem}

\begin{proof}
Suppose $\mathcal{A}$ wins with non-negligible probability
$\epsilon$. We construct reductions that contradict each of the
three assumptions, showing $\epsilon$ is bounded by the sum of the
corresponding advantages.

\emph{Step 1: extract a witness.} Since $\sigma^* = (C^*, \pi^*)$
is accepting, by knowledge soundness of the ZK argument there
exists an extractor $\mathcal{E}$ that, running $\mathcal{A}$ as a
subroutine, produces a witness $(p^*, r^*)$ such that, setting
$\varphi^* = \varphi_{p^*}(x^*)$ and $\delta^* = |\varphi^* - \tau|$:
\begin{itemize}[leftmargin=1.2em]
\item $\delta^* < \varepsilon$ (ZK-1);
\item $C^* = \mathsf{Pedersen}(\varphi^*, \delta^*, r^*)$ (ZK-2);
\item $\mathsf{pk} = \mathsf{Poseidon}(p^* \| \tau)$ (ZK-3).
\end{itemize}
The probability that extraction fails given an accepting proof is
$\mathsf{negl}_{\mathrm{KS}}(\lambda)$.

\emph{Step 2: $p^*$ equals the honest key.} The key $\mathsf{pk}$
was generated as $\mathsf{Poseidon}(p \| \tau)$ for the honest
parameters $p$. ZK-3 gives
$\mathsf{Poseidon}(p^* \| \tau) = \mathsf{Poseidon}(p \| \tau)$.
Either $p^* = p$, or $(p^* \| \tau, p \| \tau)$ is a Poseidon
collision. The latter occurs with probability at most
$\mathsf{negl}_{\mathrm{CR}}(\lambda)$ per signing query (the
adversary has $q$ chances to induce a useful collision via the
oracle). Hence $p^* = p$ except with probability
$q \cdot \mathsf{negl}_{\mathrm{CR}}(\lambda)$.

\emph{Step 3: $(\varphi^*, \delta^*)$ equals the honest values.}
Given $p^* = p$, the honest computation yields
$\varphi = \varphi_p(x^*)$ and
$\delta = |\varphi - \tau| = \delta(x^*)$. The signing oracle never
produced a commitment on $x^*$ (if it had, that signature would
already constitute a win for $\mathcal{A}$ without extraction;
absorb such wins into the KS term). Therefore $C^*$ is a
commitment produced by $\mathcal{A}$, and ZK-2 opens it to
$(\varphi^*, \delta^*)$. If $(\varphi^*, \delta^*) \ne
(\varphi, \delta)$, we have two distinct openings of $C^*$:
$(\varphi^*, \delta^*, r^*)$ from extraction and
$(\varphi, \delta, r')$ from any commitment consistent with the
honest values. Computational binding of Pedersen rules out distinct
openings except with probability
$\mathsf{negl}_{\mathrm{bind}}(\lambda)$. Hence
$\delta^* = \delta(x^*)$ except on this negligible event.

\emph{Step 4: contradiction.} Combining Steps 1 and 3:
$\delta(x^*) = \delta^* < \varepsilon$. But $\mathcal{A}$'s
winning condition requires $\delta(x^*) \ge \varepsilon$.
Contradiction.

Summing the failure probabilities across the three steps yields the
bound in the theorem statement.
\end{proof}

\begin{corollary}[PS-CMA is strictly stronger than EUF-CMA]
\label{cor:pscma-vs-eufcma}
Any PS-CMA-secure BBS scheme is also EUF-CMA secure. The converse
fails: a scheme where the signer can sign any $x$ (the EUF-CMA
default) is EUF-CMA secure but not PS-CMA secure, as the signer's
own signatures on non-compliant $x$ break the latter.
\end{corollary}

\begin{proof}
$(\Rightarrow)$ An EUF-CMA adversary $\mathcal{B}$ that forges
$(\mathsf{pk}, x^*, \sigma^*)$ on a fresh $x^*$ without the key
yields a PS-CMA adversary: without $p$, by knowledge soundness
$\mathcal{B}$ either succeeds on $\delta(x^*) < \varepsilon$ with
negligible probability (soundness itself), or must have solved the
extraction-binding chain, contradicting Theorem~\ref{thm:ps-cma}
under its assumptions.
$(\Leftarrow)$ EUF-CMA permits the key holder to sign any message.
Construct $\mathcal{A}$: query the signing oracle on any
non-compliant $x^*$; the oracle returns $\bot$, but a standard
signature scheme (e.g., ECDSA on a digest of $x^*$) would not;
direct construction yields an EUF-CMA-secure scheme that trivially
loses PS-CMA.
\end{proof}

\section{Implementation and Evaluation}
\label{sec:impl}

We instantiate BBS with the BN254 elliptic curve for all pairings
and commitments, Poseidon over the BN254 scalar field, and Groth16
for the ZK argument. A reference Circom implementation of
ZK-1, ZK-2, ZK-3 compiles to
\textbf{1{,}546 R1CS constraints} and produces proofs of
\textbf{192 bytes} (standard Groth16: $2\mathbb{G}_1 + 1\mathbb{G}_2$).
Verification performs three BN254 pairings, independent of circuit
size.

\paragraph{Prover performance.}
On Google Colab free tier (2 vCPU), a pure-JavaScript reference
prover (snarkjs) reports prove-median
$677$\,ms and verify-median $35$\,ms over 100 iterations. At the
commodity-hardware baseline of $\sim 1~\mu$s/constraint achievable
with native Rust provers (arkworks~\cite{arkworks},
rapidsnark~\cite{rapidsnark}), the 1{,}546-constraint core circuit
projects to $\sim 1.5$\,ms prove time; a 26K-constraint extended
circuit (policy predicate + anti-replay + additional side
conditions) projects to $\sim 26$--130\,ms.

\paragraph{Artifact.}
The reference circuit, input generator, validation tests, and
benchmark harness are provided as a single-file Colab runner
(\texttt{bbs\_circom\_colab.py}, 969 lines) that installs
dependencies, compiles the circuit, runs a Powers-of-Tau ceremony,
generates a Groth16 key, and produces reproducible benchmark output
in under three minutes on free-tier hardware. The artifact
establishes algorithmic consistency of the construction. It does
not provide the rapidsnark benchmark matching the native baseline;
this is in progress.

\section{Related Work}

\paragraph{Policy-based signatures~\cite{policy-sigs}.}
A certification authority issues signing keys tied to a policy; the
signer can sign only policy-compliant messages. The trust root is
the authority: a compromised authority can issue arbitrary keys.
BBS is self-committed: no authority, no revocation hierarchy, and
the policy is bound at $\mathsf{KeyGen}$ by the signer to the
signer's own future behavior.

\paragraph{Functional signatures~\cite{func-sigs}.}
A master key holder issues derived signing keys, each tied to a
function; the derived key can sign only outputs of the function.
BBS has no derivation tree: the root key is itself policy-bound.
The threat model differs: functional signatures address delegation
(principal constrains delegate); BBS addresses self-compromise
(principal constrains themselves, because an AI agent \emph{is}
the principal's delegated computation).

\paragraph{Anonymous credentials~\cite{anon-creds}.}
Users prove possession of attributes issued by an authority,
without revealing identity. The proof is about the \emph{holder's
attributes}; BBS's proof is about the \emph{signed action's
compliance}. The two are composable.

\paragraph{AI agent safety via capabilities.}
Recent work on LLM agent security (sandboxing, capability tokens,
action approval workflows) operates at the software layer. Our
contribution is orthogonal: BBS provides a cryptographic
substrate on which such systems can rest, replacing the
software-layer assumption at the signing boundary with a
mathematical one. We do not claim BBS subsumes
capability-based safety; BBS handles the outermost signing step,
beyond which all software-layer defenses remain valuable.

\paragraph{ZK in blockchains.}
zkRollups and similar systems prove execution correctness of
transactions inside a trusted execution model. BBS proves that the
signer of a transaction was, at the moment of signing, compliant
with a committed policy. The two proofs are orthogonal and
compose: a zkRollup inclusion proof over BBS-signed inputs yields a
system in which every included transaction is both executable and
policy-compliant by construction.

\section{Discussion and Future Work}

\paragraph{Beyond agent safety.}
BBS's core guarantee---that the set of signable actions is
cryptographically bounded---composes into architectural properties
beyond the compromised-agent threat model. A distributed ledger
accepting BBS signatures inherits compliance safety from per-signature
verification alone, decoupled from honest-quorum assumptions.
Pedersen's additive homomorphism enables $O(1)$ aggregate audit.
For transaction classes whose semantics are fully captured by
$\varphi$, the ZK circuit subsumes general-purpose execution. A
rate-limit extension expressible as a prev-chain traversal in
the circuit eliminates fee-based spam defense. Each of these is a
separate technical development, and we defer formal treatment to
follow-up work; the present paper establishes the primitive and
its security.

\paragraph{Efficiency of native prover.}
Matching the $\sim 1~\mu$s/constraint baseline requires a
rapidsnark or arkworks implementation of the circuit. Engineering
cost is low (the circuit is standard Groth16); the benchmark is in
progress.

\paragraph{Policy expressiveness.}
$\varphi$ is any function representable as a low-degree algebraic
circuit. Non-algebraic policies (floating-point arithmetic,
external data feeds) require either a circuit-level approximation
or a hash-bound commitment to an external value; the latter
reintroduces a trust boundary and is outside the scope of BBS's
core guarantee.

\paragraph{Leakage of $p$.}
Theorem~\ref{thm:ps-cma} assumes $p$ remains secret. A compromise
that leaks $p$ enables the adversary to produce signatures on
actions compliant with $\delta_{p}$, which may not match the
principal's intent if $p$ itself was chosen adversarially. The
standard countermeasure---key rotation, threshold key
generation---applies without modification.

\section{Conclusion}

An autonomous agent holding a BBS signing key cannot, regardless
of compromise, produce a valid signature on an action that violates
the committed policy. This is the first cryptographic bound on
autonomous agent behavior: not a software layer to be verified, not
a capability token to be checked, but the algebraic impossibility
of producing a witness. The construction is simple and
instantiable with today's tools. Its security rests on the same
assumptions as the standard ZK-SNARK stack.

The software-layer assumption---compliance holds only if enforcing
code is correct and unmodified---has been the implicit foundation
of every prior approach to agent safety. We replace it with
cryptographic impossibility.
